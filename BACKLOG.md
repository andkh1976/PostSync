# Журнал задач (Беклог) проекта PostSync

## Предложения по улучшению (Интерфейс и Безопасность)

### 1. Изоляция списка MAX-каналов (Приватность пользователей)
**Проблема:**
Текущая реализация серверной части (сущность `/api/max/chats`) возвращает общий список всех MAX-каналов, в которые добавлен бот. Это означает, что в выпадающем списке в панели управления при создании связки пользователь видит не только свои каналы MAX, но и каналы других администраторов бота (товарищей по команде).

**Предложенное решение (Ручной ввод ID):**
- **Удалить общий выпадающий список вариантов в интерфейсе панели управления.**
- Заменить его на обычное текстовое поле (аналогично тому, как это сделано для Telegram-канала), чтобы пользователь **вручную вводил ID MAX-канала**.
- Пользователь узнает нужный ID прямо внутри своего MAX-канала через команду бота `/chatid`.
- Как только связка создана (нажата кнопка "Создать связку"), канал привязывается к этому пользователю и отображается только в его личном списке активных связок. Общий кэш известных MAX-каналов на стороне пользовательского интерфейса больше не светится.

**План реализации:**
1. Отредактировать файл `frontend/index.html`:
   - Заменить блок выбора из списка `<select id="pair-max-select">` на поле числового ввода `<input type="number" id="pair-max-id">`.
   - Обновить логику создания пары `addPair()` для чтения значения из нового поля.
   - Убрать загрузку списка вызовом `/api/max/chats` из веб-приложения (дополнительная оптимизация).
2. На стороне приложения (`api.go`) можно будет отключить или удалить обработчик `/api/max/chats`, так как он больше не будет запрашиваться.

### 2. Мультиарендная (SaaS) архитектура MTProto (Ретро-синхронизация)
**Проблема:**
В данный момент механизм скачивания старой истории сообщений (MTProto) использует единую "дежурную" сессию (один номер телефона, который задается в конфигурационном файле `.env` через `TG_PHONE`). Это модель корпоративного использования (Self-Hosted). В случае коммерческого сервиса все клиенты не могут использовать один технический аккаунт, так как он не может состоять в приватных каналах всех пользователей продукта.

**Предложенное решение (Персональные сессии):**
- **Уйти от глобальной сессии:** Отдельный MTProto-клиент должен подниматься для каждой задачи на лету (в модуле `sync_worker.go`) с использованием уникальных байтов сессии (Session Bytes) того пользователя, который заказал синхронизацию.
- Для этого каждый клиент должен "привязать" свой Telegram-аккаунт через Панель управления (Mini App).

**План реализации (Архитектура SaaS):**
1. **База данных:** Создать новую таблицу `user_mtproto_sessions` для хранения сериализованной сессии `gotd/td` в базе PostgreSQL вместо локального файла `tg_session.json`.
2. **Панель управления (Mini App):**
   - Добавить раздел "Подключение аккаунта для истории".
   - Выдавать QR-код авторизации Telegram (или форму для ввода телефона и кода).
   - После успешной авторизации сохранять полученную сессию в базу данных с привязкой к ID клиента.
3. **Модуль `sync_worker`:**
   - Читать задачи на синхронизацию.
   - Извлекать из базы данных сессию клиента, запустившего задачу.
   - Временно, в рамках горутины, поднимать MTProto-клиент (`gotd/td`) с использованием этой персональной сессии.
   - Скачивать старые сообщения от лица этого клиента (поскольку он сам состоит в своем собственном, даже закрытом канале) и пересылать их в MAX.
   - После завершения задачи уничтожать инстанс клиента, высвобождая оперативную память.
