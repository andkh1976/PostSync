# План: Исправление безопасности привязки в API (api-pair-security)

## Обзор
**Цель:** 
1. Устранить уязвимость безопасности в эндпоинте `/api/channels/pair`, где любой пользователь может привязывать произвольные каналы, не доказывая, что он является администратором. Мы добавим проверки прав администратора в реальном времени как для каналов Telegram, так и для каналов Max, прежде чем позволить создать связь в базе данных.
2. Устранить критическую IDOR-уязвимость (Insecure Direct Object Reference) в эндпоинте `/api/tasks/cancel`, которая позволяет любому авторизованному пользователю отменять задачи ретроспективной синхронизации других пользователей.

## Тип проекта
**BACKEND** (Go API backend, `api.go`, `bridge.go`, `telegram.go`, `max.go`)

## Критерии успеха
1. Эндпоинт `/api/channels/pair` отклоняет привязку, если пользователь не является администратором в указанном `tg_chat_id`.
2. Эндпоинт отклоняет привязку, если пользователь не является администратором в указанном `max_chat_id`.
3. Если бот не является администратором в канале (что вызывает ошибку 403 Forbidden при запросе проверки), API корректно возвращает ошибку, предлагая пользователю сначала добавить бота в качестве администратора.
4. Старое поведение остается неизменным для корректно авторизованных администраторов.
5. Эндпоинт `/api/tasks/cancel` отклоняет запрос с `403 Forbidden`, если `task_id` не принадлежит запрашивающему пользователю (`owner.UserID`).

## Технологический стек
- **Go / Standard HTTP**
- `tgbotapi` (v5.5.1) для проверок в Telegram (`GetChatMember` / `GetChatAdministrators`)
- `maxbot-api-client-go` для проверок в Max (`GetChatAdmins`)

## Разбиение на задачи

### Задача 1: Создание хелпера для проверки администратора Telegram
**Агент:** `backend-specialist` 
**Навыки:** `api-patterns`
**Входные данные:** `telegram.go`
**Действие:** Реализовать `func (b *Bridge) CheckTgAdmin(chatID, userID int64) (bool, error)` в `telegram.go`. Эта функция должна использовать `b.tgBot.GetChatMember` для проверки, является ли `userID` администратором (Admin) или создателем (Creator) в `chatID`.
**Выходные данные:** Готовый к использованию метод-хелпер.
**Проверка:** Убедиться, что он корректно обрабатывает ошибку API "бот не является участником" без падения приложения.

### Задача 2: Создание хелпера для проверки администратора Max
**Агент:** `backend-specialist` 
**Навыки:** `api-patterns`
**Входные данные:** `max.go`
**Действие:** Реализовать `func (b *Bridge) CheckMaxAdmin(ctx context.Context, chatID, userID int64) (bool, error)` в `max.go`. Функция должна вызывать `b.maxApi.Chats.GetChatAdmins(ctx, chatID)` и проверять, находится ли `userID` в возвращенном списке.
**Выходные данные:** Готовый к использованию метод-хелпер.
**Проверка:** Убедиться, что метод корректно обрабатывает ошибки "бот не в чате" или аналогичные ошибки API.

### Задача 3: Внедрение проверок в `/api/channels/pair`
**Агент:** `backend-specialist`
**Навыки:** `api-patterns`, `secure-coding`
**Входные данные:** `api.go` (`handleAPIChannelsPair` около строки 672)
**Действие:** 
- Перед вызовом `b.repo.PairCrosspost`, вызвать `CheckTgAdmin` и `CheckMaxAdmin`.
- Если `CheckTgAdmin` завершается с ошибкой или возвращает false → вернуть `http.StatusForbidden` ("Вы не являетесь администратором в канале TG или бот не добавлен").
- Если `CheckMaxAdmin` завершается с ошибкой или возвращает false → вернуть `http.StatusForbidden` ("Вы не являетесь администратором в канале MAX или бот не добавлен").
**Выходные данные:** Модифицированный эндпоинт со строгими проверками безопасности.
**Проверка:** Отправить HTTP POST с неверными ID чатов и убедиться, что возвращается `403 Forbidden`.

### Задача 4: Устранение IDOR в `/api/tasks/cancel`
**Агент:** `backend-specialist`
**Навыки:** `secure-coding`, `api-patterns`
**Входные данные:** `repository.go`, `sqlite.go`, `postgres.go`, `api.go` (около строки 607)
**Действие:**
1. Добавить в интерфейс `Repository` метод `GetSyncTask(taskID int64) (*SyncTask, error)`.
2. Реализовать этот метод в `sqlite.go` и `postgres.go`.
3. В `api.go` (`handleAPITaskCancel`), перед вызовом `SetSyncTaskStatus`, получить задачу по `req.TaskID`.
4. Если задача не найдена или `task.UserID != owner.UserID`, вернуть `403 Forbidden`.
**Выходные данные:** Безопасный эндпоинт отмены задач.
**Проверка:** Отправить запрос на отмену чужого `task_id` и убедиться в получении `403 Forbidden`.
