# Правила работы и взаимодействия (RULES.md)

Этот файл содержит обобщенные паттерны, предпочтения и рабочие соглашения, извлеченные из истории взаимодействия, для стандартизации будущих задач.

## 1. Стиль постановки задач
- **Структурированность:** Задачи всегда разбиты на четкие блоки (`ЦЕЛЬ`, `ОГРАНИЧЕНИЯ`, `ЧТО МОЖНО МЕНЯТЬ`, `АЛГОРИТМ РАБОТЫ`, `ШАГИ`).
- **Явные границы:** Обязательное указание того, что конкретно можно и чего категорически нельзя трогать (например, "переписать только реализацию, не трогая сигнатуры" или "только проанализировать баг, код не менять").
- **Завершаемость:** Практически каждая задача содержит в себе критерий успешности ("сборка должна пройти", "тесты должны гореть зеленым", "чек-лист для ручной проверки").

## 2. Рабочие ограничения
- **Сохранение контрактов:** Сигнатуры экспортируемых функций (тип, название, аргументы) изменять строго **запрещено**, если иное явно не оговорено.
- **Стабильность зависимостей:** Запрещено добавлять новые импорты или библиотеки, если они не присутствовали в оригинальном коде (используем только то, что уже есть).
- **Сохранение функциональности:** Любые рефакторинги не должны изменять конечное поведение логики на всех возможных крайних случаях (edge cases), если только не поставлена задача исправить конкретный баг.
- **Никакой самодеятельности:** "Только диагностика" означает категорический запрет на любые правки файлов.

## 3. Принципы безопасности
- **Аудит перед изменением:** Обязательный предварительный поиск зависимостей (grep) по всему репозиторию перед любым переименованием переменных, файлов или бинарных артефактов.
- **Отказоустойчивость (Fallback):** Предпочесть дублирующие/резервные стратегии поведения системы в коде (например, чтение чистого размера из MTProto для обхода лимитов CDN).
- **Верификация компилятором:** Все изменения в Go-коде должны завершаться проверками: `go mod tidy`, компиляцией (`go build`), запуском тестов (`go test ./...`) и статическим анализатором (`go vet ./...`).
- **Покрытие крайних случаев:** При переписывании тестов обязательно проверять поведение на пустых значениях, `nil`, массивах нулевой длины и `invalid` строках.

## 4. Алгоритм работы (паттерн взаимодействия)
1. **Диагностика/Аудит** -> Прочитать логи/баги, найти все вхождения строки/ошибки в коде.
2. **План** -> Составить чек-лист файлов к изменению и описать стратегию решения (без написания кода).
3. **Согласование** -> (Опционально) Дождаться ОК на план.
4. **Безопасное редактирование** -> Внести изменения точно по оговоренным файлам.
5. **Валидация** -> Локально прогнать тесты и сборку программными тулами (go build/test).
6. **Отчет** -> Передать результат и git-команды для самостоятельного пуша пользователем.

## 5. Стиль ответов (Тон и формат)
- **Язык:** Строго русский технический.
- **Тон:** Краткий, сухой, по делу. Без воды и "маркетинговых фраз".
- **Код:** Примеры команд и логи всегда оформлять в markdown-блоки.
- **Структура:** Использовать списки, таблицы (если уместно) и четкие заголовки. 
- Команды для самостоятельного выполнения отдавать в виде готовых блоков для копипаста в терминал.

## 6. Технический стек и контекст среды
- **Языки:** Go (v1.24+), Vanilla JavaScript/HTML/CSS (Frontend).
- **Инфраструктура:** Linux-deploy (Coolify), Docker (`docker-compose`), GNU Make.
- **Базы данных:** PostgreSQL (основная), SQLite (резервная/тестовая, требует CGO).
- **Ключевые инструменты и либы:** `gotd/td` (MTProto), `Telegram Bot API`.
- **Проект по умолчанию:** `PostSync` (Двусторонний мост между Telegram и MAX).
