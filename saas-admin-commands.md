# Реализация управления пользователями SaaS (Админ-команды)

## Цель (Task)
Обеспечить возможность администратору сервиса управлять допуском пользователей (выдача подписок/триала) без перезапуска сервиса и правки `.env`.
Выбранные методы:
1. Автоматические уведомления при регистрации нового юзера.
2. Телеграм-команда `/users` для просмотра списка и `/sub_grant` для выдачи дней.

## Шаг 1: Расширение БД (repository.go, postgres.go, sqlite.go)
- Добавить метод `GrantSubscription(userID int64, days int) error` в интерфейс `Repository`.
- В `postgres.go` реализовать `UPDATE users SET subscription_end = NOW() + INTERVAL '$2 days' WHERE user_id = $1`.
- В `sqlite.go` реализовать аналогично с использованием `datetime('now', '+X days')` или просто Unix-таймстампом.

## Шаг 2: Уведомления и Команды (telegram.go)
В цикле обработки сообщений (`listenTelegram` -> `update.Message`):
- **Уведомления**: При вызове `TouchUser` для `msg.Chat.Type == "private"` проверять, есть ли у пользователя профиль (`GetUserProfile`). Если нет — после `TouchUser` вызывать `b.notifyAdmin` с приветственным сообщением.
- **Команда `/users`**: Если сообщение пишет админ (`msg.From.ID == b.cfg.AdminChatID`), запрашивать `ListUsers`, обходить в цикле `GetUserProfile` и выводить красиво отформатированный список со статусом подписки.
- **Команда `/sub_grant`**: Читать ID пользователя и количество дней, вызывать метод репозитория `GrantSubscription`.

## Шаг 3: Включение блокировки (bridge.go)
- В методе `checkAccess` удалить заглушку `return true` и раскомментировать вызов `profile.HasSubscription`.
- Добавить небольшой "костыль" для самого админа: если `userID == b.cfg.AdminChatID`, всегда `return true`, чтобы админ не мог себя заблокировать.

## Ограничения и Допущения
- Команды выводятся только в личные сообщения админу.
- Ограничение подписок начнет работать СРАЗУ после деплоя (текущие пользователи без выданной подписки потеряют доступ, им понадобится выдать триал).
