# Plan: MAX Message Splitting & Error Handling

## Проблема
Сообщения длиной более 4000 символов отклоняются API MAX (HTTP 400 validation error). В текущей реализации PostSync не разбивает длинные сообщения, а попав в очередь с фатальной 400-й ошибкой, продолжает ретраить их 20 раз, создавая нагрузку.

## Цели (Objectives)
1. Реализовать разбиение длинного текста (text/caption) на части, не превышающие 4000 символов, перед отправкой в MAX.
2. Обрабатывать HTTP 400 (и другие 4xx ошибки) как перманентные ошибки в модулях `telegram.go` и `queue.go`, чтобы предотвратить бесконечные повторные попытки (retries).

## План реализации (Implementation Steps)

### Шаг 1: Добавление функции разбиения текста `splitText`
1. Создать новую функцию `splitText(text string, maxLen int) []string` (возможно, в `api.go` или `telegram.go`).
2. Функция должна разбивать строку на куски размером не более `maxLen` с учетом длины в рунах (чтобы не разорвать символы UTF-8).
3. По возможности, разбивать по новым строкам (`\n`) или пробелам, если размер куска приближается к пределу, чтобы сообщения выглядели читаемо.

### Шаг 2: Изменения в `telegram.go` (`forwardTgToMax`)
1. При отправке сообщения без вложений (`mediaAttType == ""`): вместо одного длинного сообщения вызывать `sendMaxDirectFormatted` несколько раз в цикле по чанкам, полученным из `splitText`.
2. При отправке медиа (`mediaAttType != ""`): первый чанк отправлять вместе с вложением и токеном, а остальные (если текст длинный) – как отдельные текстовые сообщения.
3. В обработчике ошибок вокруг `b.enqueueTg2Max`: добавить "400" или `attachment.not.ready` (так как 400 это не всегда фаталити, если это "attachment.not.ready"). Важно: фильтровать именно валидационные фатальные ошибки, или ловить ошибку 400, за исключением `attachment.not.ready`.

### Шаг 3: Изменения в `queue.go` (`processQueueTg2Max`)
1. Добавить логику для перехвата 400-й ошибки как `permanent error`. Важно исключить из этого `attachment.not.ready`, если она все же может попасть в очередь (обычно она ретраится на месте в `sendMaxDirectFormatted`). Обновить условие:
   `strings.Contains(errStr, "400") && !strings.Contains(errStr, "attachment.not.ready")`

### Шаг 4: Тестирование
- Убедиться, что тесты `go test ./...` проходят.
- Написать юнит тесты на функцию `splitText` в `telegram_test.go` (или подходящем файле).
