# PostSync — Telegram & MAX Messenger Bridge

**PostSync** — это высокопроизводительный мост для пересылки и синхронизации контента между мессенджерами Telegram и MAX Messenger. 

Приложение спроектировано для бесшовного кросспостинга: оно не только перехватывает "свежие" сообщения в режиме реального времени, но и умеет мигрировать исторический контент из Telegram-каналов в MAX, вытягивая даже тяжелые вложения (видео, фото, документы) без перегрузки оперативной памяти.

## Архитектура
Приложение делится на несколько ключевых компонентов:
- **Webhook Gateway** (`api.go`, `telegram.go`, `max.go`): Слушает обновления от Telegram Bot API и MAX Bot API. Маршрутизирует сообщения и управляет командами.
- **MTProto Sync Worker** (`sync_worker.go`): Встроенный клиент Telegram на базе `gotd/td`. Используется для парсинга и скачивания исторических сообщений из каналов, на которые бот ранее не был подписан.
- **Media Streaming Engine** (`upload.go`): Ядро для потоковой (RAM-safe) перекачки файлов из серверов Telegram напрямую в MAX CDN. Специальный обходчик `customUploadToMax` формирует точный `Content-Length`, обходя ограничения CDN на *Chunked-Encoding*.
- **Database Repository** (`postgres.go`, `repository.go`): Слой транзакционной привязки сущностей `Telegram ID <-> MAX ID` для защиты от дублирования сообщений. 
- **Frontend Dashboard** (`frontend/`): Встроенный сервер, отдающий веб-интерфейс (Mini App). Включает в себя кастомный календарь для выбора дат синхронизации истории и настройки правил текста.

## Требования
- **Go**: 1.24.0+
- **База данных**: PostgreSQL (Рекомендуется) или SQLite (встроено, требуется CGO).
- **Учётные записи**:
  - Токен бота Telegram (`@BotFather`)
  - Токен бота MAX Messenger
  - `API_ID` и `API_HASH` разработчика (с `my.telegram.org`) для работы historical-синхронизации (MTProto).
- **Docker**: Опционально, для запуска через `docker-compose`.

## Установка и запуск

### Вариант 1: Локальная сборка (Требуется GNU Make)
1. Склонируйте репозиторий:
   ```bash
   git clone https://github.com/your-org/PostSync.git
   cd PostSync
   ```
2. Установите зависимости:
   ```bash
   go mod tidy
   ```
3. Соберите бинарный файл (компилируется с флагом `CGO_ENABLED=1` для поддержки SQLite):
   ```bash
   make build
   ```
4. Запустите приложение (предварительно создав файл `.env`):
   ```bash
   make run
   ```

### Вариант 2: Запуск в Docker
1. Скопируйте шаблон окружения:
   ```bash
   cp .env.example .env
   ```
2. Запустите оркестратор:
   ```bash
   docker-compose up -d
   ```

## Конфигурация (.env)
Обязательно определите следующие переменные в файле `.env` или внутри сервиса деплоя (Coolify / Github Actions):

| Переменная | Описание | Пример |
|------------|----------|--------|
| `TG_TOKEN` | Токен вашего бота в Telegram (BotFather). **(Обязательно)** | `123456:ABC-DEF1234` |
| `MAX_TOKEN` | Токен вашего бота в платформе MAX. **(Обязательно)** | `max-12345-token` |
| `DATABASE_URL` | Строка подключения к PostgreSQL. Если не задана, используется SQLite. | `postgres://user:pass@host/db?sslmode=disable` |
| `TELEGRAM_API_ID` | APP ID для подключения к Telegram (MTProto). **(Обязательно для истории)** | `1234567` |
| `TELEGRAM_API_HASH` | Хэш приложения (MTProto). **(Обязательно для истории)** | `abcdef1234567890` |
| `WEBHOOK_URL` | Базовый публичный URL вашего хоста для приёма вебхуков. | `https://bot.my-domain.com` |
| `WEBHOOK_PORT` | Локальный порт для запуска веб-сервера. | `8443` |
| `LOG_LEVEL` | Уровень детализации логов. | `info`, `debug`, `warn`, `error` |
| `ALLOWED_USERS` | Список ID (через запятую) админов, которым разрешено настраивать бота. | `123456,654321` |

## Основные функции
1. **Зеркалирование в реальном времени**: Пересылка свежих постов, опросов и текстовых сообщений из Telegram сразу в MAX.
2. **Захват истории (Historical Sync)**: Указав диапазон дат во встроенном Mini App, бот вытащит все посты за этот период из Telegram и перенесет их.
3. **Бесшовная передача медиа**: Фотографии, документы, и самое главное — тяжёлые видео скачиваются через `io.Pipe`, что не позволяет боту обрушиться от нехватки памяти (OOM), даже если пересылаются гигабайты контента.
4. **Маппинг форматирования**: Автоматический перевод Telegram-форматирования (MTProto Entities) в Markdown-стандарт MAX.
5. **Предотвращение дубликатов**: При перезапусках синхронизации бот никогда не отправит один и тот же пост дважды.

## Известные ограничения
1. Telegram Bot API имеет лимит в 20 МБ файлов на скачивание (50 МБ для локального Bot API сервера без Premium). Однако **историческая синхронизация** осуществляется через MTProto `gotd/td` в обход этого лимита — `customUploadToMax` поддерживает загрузку файлов любого размера.
2. Использование SQLite-драйвера в проекте требует наличия компилятора `C` (поэтому `CGO=1`). При сборке через Alpine Linux в docker всё настроено автоматически. Для production рекомендуется исключительно PostgreSQL.
3. Telegram Premium стикеры или защищенный копирайтом (restricted) контент каналов перенести невозможно.

## Структура проекта
```text
.
├── admin.go            # Валидация прав доступа и ролей чатов (MAX/TG)
├── api.go              # Ядро маршрутизации, регистрация вебхука Bot API
├── bridge.go           # Инфраструктурный слой, инициализация
├── format.go           # Конвертеры визуального форматирования текста (HTML/MD)
├── max.go              # Интеграция с SDK и REST MAX
├── telegram.go         # Регистрация команд, кнопок и обработка Telegram
├── upload.go           # Advanced-решатель потокового скачивания медиа "на лету"
├── sync_worker.go      # Фоновый демон (gotd/td) Historical Sync
├── postgres.go         # SQL-имплементация базы (PG)
├── sqlite.go           # SQL-имплементация базы (SQLite)
├── repository.go       # Универсальный интерфейс БД (CRUD)
├── queue.go            # Буферизация и отложенные задачи пересылки
├── main.go             # Основная точка входа
├── Makefile            # Автоматизация сборки и тестирования
├── Dockerfile          # Рецепт контейнеризации приложения
├── docker-compose.yml  # Оркестратор для контейнера с БД и приложением
├── frontend/           # Исходники веб-интерфейса Mini App (JS, CSS, календари)
└── migrations/         # Файлы `.sql` развертывания таблиц при старте
```
