# План: Аудит безопасности приложения (security-audit)

## Обзор
**Цель:** Провести комплексную проверку безопасности приложения (PostSync) с целью выявления уязвимостей, помимо тех, которые уже описаны в `PLAN-api-pair-security.md`. Определение потенциальных векторов атак, анализ кода и выработка рекомендаций по устранению.
**Почему это важно:** Для обеспечения защиты данных пользователей, предотвращения несанкционированного доступа и обеспечения соответствия стандартам безопасности.

## Тип проекта
**BACKEND** (Go API)

## Критерии успеха
1. Выполнен автоматизированный скрипт проверки безопасности и статический анализ кода на предмет уязвимостей.
2. Проведена ручная оценка безопасности API эндпоинтов (аутентификация, авторизация, валидация данных).
3. Составлен сводный отчет выявленных проблем (за исключением `/api/channels/pair`) с рекомендациями по устранению.

## Технологический стек
- Go
- Безопасность API, OWASP
- Skripting (Автоматизированные скрипты из `.agent/skills/vulnerability-scanner`)

## Файлы для анализа
- `api.go`
- `bridge.go`
- `telegram.go`, `max.go`
- Любые конфигурационные файлы или файлы с секретами

## Разбиение на задачи

### Задача 1: Автоматизированное сканирование (security_scan)
**Агент:** `security-auditor`
**Навыки:** `vulnerability-scanner`
**Входные данные:** Вся кодовая база проекта
**Действие:** Запустить встроенный скрипт сканирования (например, `python .agent/skills/vulnerability-scanner/scripts/security_scan.py .`) для выявления очевидных недостатков, жестко закодированных секретов и известных уязвимостей зависимостей.
**Выходные данные:** Отчёт скрипта сканирования.
**Проверка:** Скрипт отработал, результаты сохранены и готовы к интерпретации.

### Задача 2: Анализ логических уязвимостей в API
**Агент:** `backend-specialist`
**Навыки:** `api-patterns`
**Входные данные:** `api.go`, обработчики запросов
**Действие:** Ручной аудит эндпоинтов. Поиск IDOR, проблем с валидацией входных данных, внедрений, состояния гонки (race conditions) и отсутствующих проверок привилегий (не включая проблему, уже решаемую в `PLAN-api-pair-security`).
**Выходные данные:** Список найденных логических уязвимостей `backend-specialist`.
**Проверка:** Каждая проблема описана с указанием файла и строки.

### Задача 3: Симуляция векторов атаки
**Агент:** `penetration-tester`
**Навыки:** `red-team-tactics`
**Входные данные:** Логика аутентификации и внешние взаимодействия.
**Действие:** Оценка механизмов доверия внешним сервисам (ботам Telegram, Max), проверки CSRF, CORS, манипуляций с сессиями.
**Выходные данные:** Отчет о тестировании на проникновение.
**Проверка:** Подготовлены векторы атаки и стратегии смягчения (mitigation).
